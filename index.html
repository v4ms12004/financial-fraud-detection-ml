01 · About

The Problem &
The Project

Sponsored by the Industry Advisory Board through KDSC's Community Data Labs, this project tackles one of the most challenging problems in financial technology: detecting fraud in a sea of legitimate transactions where fraud represents just 0.13% of all activity.

Sponsor

Industry Advisory Board

Real-world applied project under the Kansas Data Science Consortium framework, building production-grade fraud detection infrastructure.

Core Challenge

Extreme Class Imbalance

With fraud at 0.13% of all transactions, standard accuracy metrics are completely meaningless. A model predicting "not fraud" every time achieves 99.87% accuracy.

Research Question

Time-Aware Detection

Can leakage-safe, time-aware tree-based models detect rare fraud under distribution shift — and do they learn real fraud signals or just simulator rules?

04 · Methodology

The ML Pipeline

Three critical design decisions separate this pipeline from naive approaches: temporal integrity, behavioral feature engineering, and class imbalance handling.

Problem
Severe Class Imbalance
Only 0.13% of transactions are fraud. Standard accuracy is meaningless. Majority-class classifier achieves 99.87%.
Solution
AUPRC + scale_pos_weight
Switch evaluation to AUPRC (Area Under Precision-Recall Curve). Set XGBoost scale_pos_weight = #non-fraud / #fraud ≈ 773.
Problem
Data Leakage Risk
Random train/test splits allow future transactions to leak into training. Cumulative features computed on full data leak future fraud signals.
Solution
Strict Temporal Ordering
80/20 chronological split. All panel features use past-only data (cumsum minus current). TimeSeriesSplit cross-validation preserves time ordering.
Problem
Raw Features Lack Context
Account IDs are useless as strings. Raw transaction amounts don't capture behavioral anomaly — a $10k transfer is normal for one account, suspicious for another.
Solution
Panel Behavioral Features
Engineered 15+ features capturing account history: transaction count, cumulative amount, average amount, tenure, time since last transaction — for both sender and receiver.

The Cost of Data Leakage

Time-Sorted vs
Random Split

A direct experiment answering: how much does proper temporal ordering matter? Two identical models — same features, same hyperparameters — evaluated with different splitting strategies.

AUPRC — Time-Sorted vs Random Split
✓ Time-Sorted (Our Approach)
AUPRC = 0.9958  |  Recall = 98.8%
Train on first 80% of time steps (3,959 fraud examples), test on last 20% (4,254 fraud). Simulates real deployment: the model predicts on genuinely unseen future transactions. This is the honest, realistic performance estimate.
✗ Random Split (Naive Baseline)
AUPRC = 0.9939  |  Recall = 97.6%
Random 80/20 stratified split (6,570 fraud in training, 1,643 in test). Despite having nearly double the fraud training examples, this model performs worse — revealing that temporal structure matters more than raw fraud volume.

Unexpected finding: the time-sorted model outperformed the random split model despite training on 40% fewer fraud examples (3,959 vs 6,570). Fraud in PaySim is temporally concentrated — later steps have higher fraud density. The time-sorted model learns from the most recent fraud patterns, making it better at predicting future fraud. This confirms temporal ordering is not just methodologically correct — it actively produces a stronger model.

07 · Feature Importance

What the Model
Actually Learned

XGBoost's feature importance reveals which signals drive fraud detection — and this analysis directly informed our generalizability experiment design.

Top Features by XGBoost Importance Score

Critical insight: The top three features — dest_tenure (38.5%), orig_txn_count_prev (27.7%), and dest_txn_count_prev (8.7%) — account for 75% of all model decisions. These panel behavioral features capture the fact that mule accounts are brand new (zero tenure, zero history) while victim accounts are active. Balance-based features like delta_org and amount_over_oldorg contribute only ~1.5% combined, meaning the complications we added to break those features were attacking the wrong signals entirely.

07 · Generalizability Experiment

Does the Model Learn
Real Fraud — or PaySim?

A critical open question raised during project discussions: since PaySim uses deterministic agent-based simulation logic, could the model be learning PaySim's internal rules rather than generalizable fraud behavior? We designed a novel experiment to test this.

Experimental Design: Build three custom Python transaction simulators with identical fraud patterns (same TRANSFER→CASH_OUT mechanism) but with different statistical distributions, account ID ranges, balance distributions, and random seeds. Train model on PaySim, test on each synthetic dataset, measure AUPRC. If the model learned real fraud signals, AUPRC should remain high.

Experiment 1 — Clean Synthetic Dataset
AUPRC = 1.000
Same fraud pattern as PaySim but with fresh account IDs (C9000000+), different balance distribution (mean $250K vs PaySim's $833K), different amount distribution, 600 steps instead of 743, and different random seed. Result: perfect 1.0 AUPRC with zero false positives and zero false negatives (263 fraud cases). The model generalizes well across different statistical distributions.
AUPRC 1.000 Different Distributions Same Fraud Pattern
Experiment 2 — Realistic Synthetic (7 Complications)
AUPRC = 1.000
Added five real-world complications: (1) partial balance drains (60-95% instead of 100%), (2) balance noise from fees/rounding, (3) split fraud transactions across 2-3 steps, (4) legitimate large transfers resembling fraud, (5) normal accounts occasionally hitting zero balance. All five complications were designed to break the balance-based features. Result: still perfect 1.0 — because the complications targeted the wrong features. The model relies 75% on tenure/count signals, not balance features.
5 Complications Added AUPRC 1.000 Wrong Features Targeted
Experiment 3 — Advanced Synthetic (Tenure-Aware, 7 Complications)
30% Window: AUPRC = 0.003
After feature importance analysis revealed tenure/count features dominate at 75%, we redesigned the generator: mule accounts now receive 10-50 normal warm-up transactions before fraud occurs (defeating dest_tenure and dest_txn_count_prev), and victims are selected from the most active accounts (defeating orig_txn_count_prev). Overall AUPRC: still 1.0. But temporal sensitivity exposed something fascinating — at the 30% training window, AUPRC collapsed to 0.003 with Recall=1.0 and Precision=0.003. The model had never seen fraud in early training and broke completely. At 60%+ it snapped back to perfect. This reveals a training data threshold effect: the model needs sufficient fraud examples to learn the pattern, but once it does, even heavily disguised fraud is detectable.

08 · Key Findings

What We Learned

Six substantive findings that go beyond model performance numbers, each with implications for how fraud detection systems should be designed and evaluated.

01

Temporal Evaluation is Non-Negotiable

Random train/test splits inflate performance by allowing future fraud patterns to leak into training. Time-aware 80/20 chronological splits with TimeSeriesSplit cross-validation are essential for realistic evaluation. Models evaluated with random splits should not be trusted for fraud detection deployment.

02

Behavioral Panel Features Dominate Balance Features

Despite intuition suggesting balance-based features (delta_org, amount_over_oldorg) would be most informative, feature importance revealed that account tenure and transaction count history account for 75% of decisions. Behavioral context — how unusual this transaction is relative to account history — matters far more than the raw transaction amounts.

03

XGBoost Outperforms Random Forest Under Temporal Stress

Both models achieved high baseline AUPRC (~0.995–0.998), but Random Forest showed greater instability under strict temporal splits and distribution shift. XGBoost's sequential error correction and direct class imbalance handling via scale_pos_weight gave it consistent temporal stability across all validation windows.

04

The Model Generalizes Across Statistical Distributions

Testing on three custom synthetic datasets with different balance distributions, account ranges, and random seeds showed AUPRC remaining at 1.0 overall. The core fraud signals — account behavioral anomalies, tenure patterns, and TRANSFER→CASH_OUT sequences — are robust to changes in the underlying statistical distributions of the data.

05

A Training Data Threshold Effect Exists

The 30% training window experiment revealed a critical threshold: with insufficient historical fraud examples, the model fails catastrophically (AUPRC 0.003, essentially inverted scores) despite achieving perfect performance with more data. This suggests fraud detection systems deployed in new markets need a minimum operational period before the model can be trusted.

06

AUPRC is the Only Valid Metric for Rare Events

With 0.13% fraud rate, a model predicting "not fraud" every time achieves 99.87% accuracy. ROC-AUC is similarly misleading due to the massive true-negative count. AUPRC evaluates the precision-recall tradeoff specifically for the rare positive class, making it the only appropriate primary metric for extreme class imbalance problems.
